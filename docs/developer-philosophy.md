# EnShift Developer Philosophy

> システムを管理するのではなく、人を支えるためのシステムを作る。

## 私たちが支える人

EnShiftを毎日使うのは、子どもたちと向き合う保育園・認定こども園の先生です。
先生が画面の操作に時間や気持ちを取られず、本来の仕事へ安心して戻れることを、機能数より優先します。

## 判断の基準

設計、文言、実装、レビューでは、次の順に問いかけます。

1. 説明書を読まなくても、次に何をすればよいか分かるか。
2. 押し間違いや思い込みを防げるか。
3. 操作した結果と、次に起きることが日本語で伝わるか。
4. 忙しい時間や小さな画面でも、安心して操作できるか。
5. 管理の都合ではなく、先生の仕事を軽くできているか。

## UIと言葉

- 今日必要な情報を最初に見せ、管理機能は一段下げる。
- 1画面の主操作を原則1つにし、ボタンには結果が分かる動詞を書く。
- エラーコードや技術用語ではなく、起きたことと次の行動をやさしい日本語で伝える。
- 空の画面にも、何もない理由と次にできる操作を示す。
- 色、形、アイコンだけに頼らず、文字でも状態を伝える。
- 待ち時間、通信切断、保存完了を隠さず、利用者が不安にならない表示を続ける。

## 安全と品質

- 園・利用者・権限の境界をすべての処理で守る。
- 保存前の確認、処理中の二重操作防止、完了後のフィードバックを一組として設計する。
- 既存機能を守り、必要最小限の変更を小さく検証する。
- Console Error／Warning、横スクロール、行き止まりの空画面をリリース前に確認する。
- 個人情報、秘密情報、外部サービスへの送信を安易に増やさない。

## 完了の定義

コードが動くだけでは完了ではありません。
先生が迷わず操作でき、失敗しても戻り方が分かり、安心して一日の仕事を続けられる状態を完成とします。
